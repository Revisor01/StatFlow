The root cause: `loadAllAccountsData()` calls `setActiveAccount()` in a loop for each account, which:
- Changes the global `activeAccount` (confusing if user interacts mid-loop)
- Triggers `updateWidgetCredentials()` + `reloadAllTimelines()` for EACH iteration
- Widget may refresh with mid-loop account credentials

**Fix Strategy — Two options depending on what the code allows:**

**Option A (preferred): Pass credentials directly to API actor methods**

If `UmamiAPI` and `PlausibleAPI` have or can get methods that accept credentials as parameters (e.g., `reconfigure(serverURL:token:)` or similar), create a lightweight method in AccountManager:

```swift
/// Temporarily configure provider for a specific account WITHOUT switching global state
func configureProviderForAccount(_ account: AnalyticsAccount) async {
    // Configure the API actor with this account's credentials from Keychain
    // But do NOT: update activeAccount, post notifications, or reload widgets
    let credentials = loadCredentialsFromKeychain(for: account)
    switch account.providerType {
    case .umami:
        await UmamiAPI.shared.reconfigure(serverURL: account.serverURL, token: credentials.token)
    case .plausible:
        await PlausibleAPI.shared.reconfigure(serverURL: account.serverURL, apiKey: credentials.apiKey)
    }
    AnalyticsManager.shared.setProvider(for: account.providerType)
}
```

Then rewrite `loadAllAccountsData()`:
```swift
func loadAllAccountsData() async {
    let originalAccount = AccountManager.shared.activeAccount
    for account in accounts {
        // Configure API without global side effects
        await AccountManager.shared.configureProviderForAccount(account)
        // ... load data ...
    }
    // Restore original — use full setActiveAccount ONLY here (once, at the end)
    if let original = originalAccount {
        await AccountManager.shared.setActiveAccount(original)
    }
}
```

**Option B (simpler fallback): Suppress widget reload during multi-account loop**

Add a flag to AccountManager:
```swift
var suppressWidgetReload = false
```

In `applyAccountCredentials`, wrap the widget reload:
```swift
if !suppressWidgetReload {
    updateWidgetCredentials()
    WidgetCenter.shared.reloadAllTimelines()
}
```

In `loadAllAccountsData`:
```swift
AccountManager.shared.suppressWidgetReload = true
defer { AccountManager.shared.suppressWidgetReload = false }
for account in accounts {
    await AccountManager.shared.setActiveAccount(account)
    // ... load data ...
}
if let original = originalAccount {
    AccountManager.shared.suppressWidgetReload = false  // re-enable before final restore
    await AccountManager.shared.setActiveAccount(original)
}
```

**Choose Option A if the API actors already have a `reconfigure` method that accepts credentials directly. Choose Option B if reconfiguring without Keychain is not straightforward.**

Read the actual API actor code to determine which option fits. Either way, the result must be:
- Widget reloads happen AT MOST ONCE after the loop completes (when restoring original account)
- Global `activeAccount` is not changed during the loop OR is clearly locked/flagged
- No `.accountDidChange` notifications are posted during the loop iterations

**Additional fix in AccountManager.updateWidgetCredentials():**
Ensure the file write uses `.atomic` write option (likely already does — verify) and that `WidgetCenter.shared.reloadAllTimelines()` is called AFTER the write completes (not in a detached task).

   ```swift
   func refresh() async {

           // ... more property writes, each preceded by guard if after an await

       }

   func loadDetail() async {

           defer { if !Task.isCancelled { isLoading = false } }
           do {
               let result = try await ...

   ```
