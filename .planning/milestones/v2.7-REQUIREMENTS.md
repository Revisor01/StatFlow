# v2.7 Requirements — Stability & Architecture

**Milestone:** v2.7 Stability & Architecture
**Goal:** Alle aktiven Bugs fixen, Task-Cancellation flächendeckend einbauen, Cache-Strategie auf Offline-Only umstellen, Tech Debt systematisch abbauen.
**Source:** CONCERNS.md Audit vom 2026-04-04 (nach 11-Bug-Fix-Session)

## Leitsatz Cache

> Cache dient **ausschließlich** als Offline-Fallback. Wenn das Gerät online ist, werden **immer** frische Daten von der API geholt. Der Cache wird nach erfolgreichen API-Calls aktualisiert, aber nie als primäre Datenquelle angezeigt.

## Requirements

### BUG-01: CompareChartSection @ObservedObject Fix
- **Priorität:** Kritisch
- **Problem:** `CompareChartSection` verwendet `let viewModel` statt `@ObservedObject var viewModel` — Chart aktualisiert sich nicht bei Datenänderungen
- **Scope:** `CompareChartSection.swift` + Audit aller Child-Views die ein ObservableObject empfangen
- **Akzeptanzkriterien:**
  - Alle Child-Views die ein ObservableObject empfangen verwenden `@ObservedObject`
  - Kein einziges `let viewModel: *ViewModel` mehr im Projekt (außer in Previews)
  - CompareChart aktualisiert sich sofort wenn Parameter geändert werden

### BUG-02: Cache nur für Offline
- **Priorität:** Kritisch
- **Problem:** Cache-Daten werden als Vorschau angezeigt und überschreiben manchmal frische API-Daten. Dashboard zeigte stundenlang 147 statt 213 Besucher.
- **Scope:** `AnalyticsCacheService`, `DashboardView` (ViewModel), `WebsiteDetailViewModel`
- **Akzeptanzkriterien:**
  - Wenn online: API-Daten werden **immer** direkt geladen, Cache wird **nicht** vorgeladen
  - Wenn offline (kein Netz): Cache-Daten werden als Fallback angezeigt mit visuellem Hinweis ("Offline — letzte Daten von [Zeitpunkt]")
  - Cache wird nach jedem erfolgreichen API-Call aktualisiert (für nächsten Offline-Fall)
  - Cache-TTL wird bei Offline-Anzeige respektiert — Daten älter als 24h werden nicht gezeigt
  - Website-Liste-Cache prüft `isExpired` (aktuell fehlt das)
  - `clearStaleEntries()` wird beim App-Start aufgerufen
  - `evictOldestEntries()` wird aufgerufen wenn Cache > 50MB

### BUG-03: Widget Account-Sync Race Condition
- **Priorität:** Hoch
- **Problem:** Widget zeigt manchmal Daten des alten Accounts nach Wechsel
- **Scope:** `AccountManager.swift`, `SharedCredentials.swift`
- **Akzeptanzkriterien:**
  - Widget-Refresh wartet auf abgeschlossene Credential-Anwendung
  - Kein Flash von alten Account-Daten nach Wechsel

### TASK-01: Task-Cancellation in allen ViewModels
- **Priorität:** Kritisch
- **Problem:** Nur `WebsiteDetailViewModel` cancelt vorherige Loads. 7+ andere ViewModels erlauben parallele Loads die sich gegenseitig überschreiben.
- **Scope:** Alle ViewModels: Events, Reports, Sessions, SessionDetail, Journey, Retention, Pages, Compare, Dashboard, Realtime, Insights
- **Akzeptanzkriterien:**
  - Jedes ViewModel mit async Load hat ein `loadingTask`-Handle
  - `loadingTask?.cancel()` wird vor jedem neuen Load aufgerufen
  - `guard !Task.isCancelled else { return }` vor jedem State-Write
  - Kein Überschreiben von gültigen Daten durch gecancelte Requests

### TASK-02: Account-Switch ohne globalen Singleton-State
- **Priorität:** Hoch
- **Problem:** `loadAllAccountsData()` wechselt den aktiven Account temporär für jeden Account — User-Interaktion währenddessen sieht falschen Account
- **Scope:** `DashboardView` (ViewModel), `AccountManager`
- **Akzeptanzkriterien:**
  - Credentials werden direkt an API-Calls übergeben statt globalen State zu wechseln
  - Oder: Expliziter Loading-State der Interaktion blockiert während Account-Switch läuft

### REFACTOR-01: Network-Error-Detection deduplizieren
- **Priorität:** Mittel
- **Problem:** Identischer 5-Zeilen-Block in 5 ViewModels copy-pasted
- **Scope:** Extension auf `Error` oder Utility-Funktion
- **Akzeptanzkriterien:**
  - Eine einzige `isNetworkError`-Property/Funktion
  - Alle 5 Stellen verwenden sie
  - Neue Error-Codes (z.B. `.dataNotAllowed`) müssen nur an einer Stelle ergänzt werden

### REFACTOR-02: DateFormatter-Instanzen wiederverwenden
- **Priorität:** Mittel
- **Problem:** `ISO8601DateFormatter()` und `DateFormatter()` werden in Loops und Closures neu erstellt — teuer bei 365 Datenpunkten
- **Scope:** 7+ Dateien mit Formatter-Allocations in Hot Paths
- **Akzeptanzkriterien:**
  - Shared `static let` Formatter als Properties
  - Keine Formatter-Allokation innerhalb von Loops oder `.map {}`-Closures
  - Messbare Verbesserung bei Charts mit vielen Datenpunkten

### REFACTOR-03: ViewModels in eigene Dateien extrahieren
- **Priorität:** Mittel
- **Problem:** 7 Dateien > 600 Zeilen mit embedded ViewModels — schwer zu testen und zu warten
- **Scope:** DashboardView (1189 LOC), SessionsView (706), RealtimeView (650), SettingsView (742)
- **Akzeptanzkriterien:**
  - Jedes ViewModel in eigener Datei
  - Keine Datei > 500 Zeilen (außer API-Services)
  - Bestehende Tests laufen weiterhin

### REFACTOR-04: Dependency Injection für ViewModels
- **Priorität:** Niedrig
- **Problem:** Alle ViewModels referenzieren `UmamiAPI.shared` / `PlausibleAPI.shared` direkt — untestbar
- **Scope:** Alle ViewModels
- **Akzeptanzkriterien:**
  - ViewModels akzeptieren `AnalyticsProvider` via init-Parameter mit Default-Wert
  - Tests können Mock-Provider injizieren
  - Bestehender Code bleibt unverändert (Default-Wert = `.shared`)

### REFACTOR-05: LazyVStack-Audit
- **Priorität:** Mittel
- **Problem:** LazyVStack mit conditional Content in 5+ Views — dasselbe Risiko wie der heute gefixte Bug
- **Scope:** Dashboard, ReportsHub, Events, Sessions, Admin Views
- **Akzeptanzkriterien:**
  - LazyVStack nur für homogene Listen mit > 50 Items
  - Alle Views mit conditional Content (if/else auf ViewModel-State) verwenden VStack
  - Dokumentierter Kommentar wo LazyVStack bewusst verwendet wird

### REFACTOR-06: Strukturiertes Logging
- **Priorität:** Niedrig
- **Problem:** 88 `print()`-Statements — Debug-Rauschen, keine Filterbarkeit
- **Scope:** Alle `#if DEBUG print(...)` Stellen
- **Akzeptanzkriterien:**
  - Logging via `os.Logger` mit Subsystem und Category
  - Kategorien: `api`, `cache`, `auth`, `ui`
  - `print()`-Statements entfernt
  - Logs in Console.app filterbar

### SEC-01: KeychainService Force Unwraps entfernen
- **Priorität:** Niedrig
- **Problem:** `value.data(using: .utf8)!` kann bei Non-UTF-8 Strings crashen
- **Scope:** `KeychainService.swift` (Zeilen 18, 89)
- **Akzeptanzkriterien:**
  - `guard let` statt Force Unwrap
  - Error wird geworfen statt Crash
