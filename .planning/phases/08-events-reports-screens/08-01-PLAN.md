**EventsView** (Hauptliste):
- `let website: Website`
- `@StateObject private var viewModel: EventsViewModel`
- `@State private var selectedDateRange: DateRange = .thisWeek`
- `@State private var selectedEvent: MetricItem?` (fuer Detail-Navigation)

Layout (wie SessionsView):
1. DateRange-Picker oben (ScrollView horizontal mit DateRangeChip, exakt wie in SessionsView)
2. Stats-Header: GlassCard mit eventStats (Events-Count, Properties-Count, Records-Count) als kompakte HStack
3. Event-Liste als ScrollView > LazyVStack:
   - Jede Zeile: Event-Name links, Count rechts als Badge, chevron.right
   - Tap setzt selectedEvent und navigiert per NavigationLink zu EventDetailView
4. Empty State: ContentUnavailableView mit "calendar.badge.exclamationmark" Icon wenn events leer und nicht isLoading
5. Loading State: ProgressView wenn isLoading und events leer

Navigation: `.navigationTitle(String(localized: "events.title"))`, `.navigationBarTitleDisplayMode(.inline)`
Task: `.task { await viewModel.loadEvents(dateRange: selectedDateRange) }`
onChange: Wie in SessionsView — bei DateRange-Aenderung neu laden
Refreshable: `.refreshable { await viewModel.loadEvents(dateRange: selectedDateRange) }`

**EventDetailView** (als separate struct im selben File):
- Parameter: `website: Website`, `eventName: String`, `dateRange: DateRange`
- Eigener @StateObject EventsViewModel (oder den uebergebenen nutzen)

1. Header-GlassCard: Event-Name gross, Stats (Total Count aus der Liste)
