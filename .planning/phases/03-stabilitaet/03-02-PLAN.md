**AccountManager.swift — Zeile 295: asyncAfter entfernen**

Das `DispatchQueue.main.asyncAfter(deadline: .now() + 0.3)` in `applyAccountCredentials()` durch synchrones Posting ersetzen. Die Methode ist `@MainActor` — alle vorherigen Property-Zuweisungen (Keychain, setSitesWithoutPersist, reconfigureFromKeychain, setProvider) sind synchron und abgeschlossen bevor die Notification-Zeile erreicht wird.

VORHER:
```swift
DispatchQueue.main.asyncAfter(deadline: .now() + 0.3) {
    NotificationCenter.default.post(name: .accountDidChange, object: nil)
}
```

NACHHER:
```swift
NotificationCenter.default.post(name: .accountDidChange, object: nil)
```

Einfach den asyncAfter-Block durch den direkten Post-Aufruf ersetzen. Kein Task.yield(), keine Ersatz-Delays.

**AuthManager.swift — Zeile 93: Task.sleep entfernen**

In `loadPlausibleCredentials()` die `try? await Task.sleep(nanoseconds: 100_000_000)` Zeile loeschen. Der Task-Block bleibt erhalten, nur das Sleep wird entfernt.

VORHER:
```swift
Task { @MainActor in
    try? await Task.sleep(nanoseconds: 100_000_000) // 0.1 seconds
    let sites = PlausibleSitesManager.shared.getSites()
    ...
}
```

NACHHER:
```swift
Task { @MainActor in
    let sites = PlausibleSitesManager.shared.getSites()
    ...
}
```

Begruendung: PlausibleSitesManager.shared ist ein lazy Singleton das in AccountManager.init() (frueher in der Initialisierungskette) bereits initialisiert wird. getSites() ist synchron und gibt sofort zurueck.

WICHTIG: Nur die zwei Timing-Hack-Zeilen aendern. Keine Refactorings, keine neuen Patterns. Die umliegende Logik bleibt unveraendert.
